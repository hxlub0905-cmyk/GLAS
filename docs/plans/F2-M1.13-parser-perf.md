## Goal & Context

### 為什麼做

M1.12 端到端跑通後實測（D2DB E3B_CMG_CMP_D2DB_250930.oas, 345 MB, layer 17/101, 5M records）：

```
store.run()      77.7s
walk_to_root    114.0s
─────────────────────
total          ~192s   for 5M records (0.69% of full file)
```

外推全檔約 **3.6 小時**。Partial slice 已可用，但：
1. 每次重開同一份 GDS 都要重跑 192s — 設計迭代痛苦
2. 全檔 production batch 跑不動，主程式整合 (F3) 卡這道
3. Walker 比 store.run 還慢 → repetition expansion 是 Python loop hot path

### 想達成什麼

| 場景 | 現況 | 目標 |
|---|---|---|
| 同一 GDS+layer 第二次開 | 192s (5M partial) / 3.6 hr (full) | **< 5 秒**（cache 命中）|
| 第一次 partial 5M records | 192s | **< 60 秒**（3× 提升）|
| 第一次 full D2DB load | ~3.6 hr | **< 30 分鐘**（7× 提升）|

### 跟現有系統的關係

- **不動 GUI/Recipe**：純 backend 加速，`oasis_streamer / oasis_store / oasis_walker` 三層 API 表面盡量保留
- **不動 153 既有 pytest**：所有改動需 backward-compatible（caching opt-in；vectorize 對 caller 透明；fast-consumer 為新 API、舊 `iter_records` 留著）
- **不引入必要重型依賴**：numpy 已有；不加 numba/cython/cffi 為**必要**依賴（M1.13.4 C ext 視需要、且設計為 optional fast-path）

## Q&A Decisions

### Q1: M1.13 第一個子項？
**選項：** caching / walker vec / fast-consumer / C ext
**選擇：** **caching (子項 4)**
**理由：** 最低風險、最高 UX 收益（第二次秒開），不動 parser API；使用者可立刻享受迭代加速。

### Q2: Cache 存哪、何時 invalidate？
**選擇：** **暫不決定，先設計 API 再選**（M1.13.1 第一個 subtask 即為 Q&A spec）

### Q3: M1.13 是否拆獨立 plan？
**選擇：** **拆 `docs/plans/F2-M1.13-parser-perf.md`**（本檔）
**理由：** 4 個子項各自有獨立 design space + risk，集中在 F2 主 plan 會太長；獨立 plan 也方便日後 F2 整合階段引用。

### Q4: 成功標準？
**選擇（user 全勾）：**
- 300 MB D2DB full-load < 30 min
- Cache 命中後 < 5 秒
- Partial 5M records < 60 秒
- 既有 153 pytest 全綠 + 新增 perf regression test

### Q5: C extension 在 plan 的定位？
**選擇：** **列為 M1.13 最後一個子項，做完前 3 子項再評估是否需要**
**理由：** 若 caching + walker vec + fast-consumer 已達 30 min / 5s / 60s 三標準，C ext 視為 nice-to-have 推遲到 M1.14。

### Q6: Walker vectorize 手法？
**選擇：** **Pure numpy broadcasting**（依 Claude 建議）
**理由：** 不加 optional dep、足夠達 5–10× 提升；DFS 的 control flow 留 Python，hot path 只 vectorize 內層 corner-transform + repetition fan-out。

### Q7: Perf regression test 載體？
**選擇：** **`tests/test_perf_oasis.py` + `@pytest.mark.slow` marker**（依 Claude 建議）
**理由：** 一致 pytest 慣例；用 `pytest -m "not slow"` 預設跳過、CI/快驗時不卡；user 帶規格檔手動觸發；`pytest.skip` 找不到大檔時優雅跳過。

### M1.13.1: Cache layer (file_hash + layer_set → walked ndarrays)  [status: done (2026-05-15, user verified on 345 MB Calibre D2DB; cache HIT 0.33s ≪ 5s target)]

讓「同一 GDS + 同一 layer set」第二次開直接從 cache 載 walker output，目標 < 5 秒。

**Cache spec Q&A 決議（2026-05-15）：**
- 存哪：**`~/.mmh/oasis_cache/`**（與 recipes/calibrations 同層；env var `MMH_OASIS_CACHE_DIR` 可 override）
- Hash 策略：**sha256 + layer_set + max_records**（300 MB 約 1–3 秒一次性開銷，cache hit 不受影響）
- 序列化：**`np.savez_compressed`**（per-layer pts/offs/bbs + JSON manifest in `meta` uint8 array）
- Eviction：**手動清理 only**（CLI `oasis_cache.py purge` / `invalidate <key>`）
- Schema versioning：`SCHEMA_VERSION = 1`，存進 key payload + manifest；bump 即 invalidate 全部舊 entry

### M1.13.2: Walker vectorize (numpy broadcasting on hot path)  [status: done (2026-05-15, user verified on D2DB partial 5M)]

**User runtime 驗證（2026-05-15, D2DB 345 MB partial 5M, layer 17/101, cache MISS）：**

```
[gds-loader] 77.7s  store.run() done; 45,745 raw rectangles
[gds-loader] 77.7s  walk_to_root L17/D101 starting
[gds-loader] 80.1s  layer L17/D101: 45,745 rect -> 45,745 polys flattened
[gds-loader] 80.3s  cache SAVED 0.7 MB
```

**結論：**
- **walk_to_root = 2.4 秒**（80.1 − 77.7）vs M1.13.1 baseline 114 s = **47× speedup**
- 45,745 rect emit 跟 M1.13.1 結果**完全一致**（bit-identical 保證成立）
- 第一次開總 80.3s（store 77.7s + walk 2.4s + cache save 0.2s），下次 cache HIT 仍 0.33s
- store.run() 仍 77.7s（M1.13.1 路徑未動，下個瓶頸）→ M1.13.3 fast-consumer / M1.13.4 C ext 主攻 store

把 `apply_to_rects` 對 N 個 rect 內 4 corner 的 transform 完全 vectorize；把 `walk_to_root` 中對 K 個 repetition_offsets 的 fan-out 改成一次 broadcast 而非 K iter。

**Hot path 確認：** `Transform.apply_to_rects` 自 M1.11c 起對 N 已 vectorize；真正瓶頸是 `_expand_placement` 的 K-repetition Python loop（user D2DB 1,160 placements × avg 3,360 reps = 3.9M iters），每 iter 都重建 Transform + compose + recurse。

**實作（已完成）：**

- [x] **Hoist invariant transform**：rotation/flip/mag 在 K loop 外建一次（同個 placement 的 K reps 共享同個 M）
- [x] **Vectorized translation composition**：`(K, 2)` offsets → 一次算出 `(K, 2)` composed translations，省 K 次 numpy matmul/add
- [x] **Leaf fast path**：target 沒有 nested placement 且不在 cycle → 一次 batch emit `K * N` rect、**跳過 K 次遞迴**
  - 新 helper `_batch_transform_rects(rects, M, translations)`：先對 N corners 套 M（一次），算 per-rect bbox，再 broadcast `(K,)` translations → 輸出 `(K*N, 4)`
  - 等價公式：`floor(min(M@c) + t) == floor(min(M@c + t))`（min 跟 const t 可交換）→ bit-identical 跟既有 `apply_to_rects` K 次 concat
- [x] **Slow path** (target 有子 placement 或 cycle)：仍 K 次遞迴，但 transform 用已 batch 算好的 composed_M/composed_ts（不再 per-K 重建 from_placement）
- [x] Polygon path：M 一次套、K 次 translate-and-append（polygon 在 D2DB 是 0.02%，不再多優化）
- [x] Arbitrary angle：之前 per-K warn K 次 + skip K 次；現在 warn 一次 + `arbitrary_angle_skipped += K`（counter 語意不變、tests 鎖住）
- [x] Public API 未動 (`walk_to_root` / `walk_polygons_to_root` 簽名一致)；`WalkStats` 計數語意一致（cells_visited / placements_expanded / rectangles_emitted 均跟 M1.11c 數量等同）
- [x] 新 7 個 `TestVectorizeEquivalence` test：K=500 leaf fast path / slow path with nested placement / slow path with rotation / arbitrary angle stats / K=1 empty offsets / `_batch_transform_rects` K=1 matches `apply_to_rects` / K=3 broadcast 正確
- [x] 既有 31 walker test 全保留通過（含 TestRepetitionExpansion 的 K=3 leaf 跟 K=2 with rotation）

**驗證（sandbox 已過）：**
- [x] `pytest tests/test_oasis_walker.py -v` → **38 passed**（31 既有 + 7 新 TestVectorizeEquivalence）
- [x] `pytest tests/test_oasis_streamer.py tests/test_oasis_store.py tests/test_oasis_walker.py tests/test_oasis_cache.py -q` → **188 passed**
- [x] `python3 -m py_compile tools/oasis_walker.py` ✓
- [x] **Sandbox microbench**（synthetic D2DB-scale）：1,160 placements × 3,360 reps × 1 rect → 3,897,600 emitted rect **1.47 秒** total。對 user 規格機 150–378s 估 100×+ 加速

**待 user runtime 驗證：**
1. `python tools/oasis_cache.py purge` 清掉 M1.13.1 快取
2. 用 layer 17/101 + max_records=5M 開 D2DB → walk_to_root 階段預期 **< 30s**（vs M1.13.1 baseline 150–378s）

### M1.13.3: Fast-consumer + byte-reader 重構  [status: done (2026-05-20, user verified D2DB partial 5M: store 36.2s = 2.15× vs baseline)]

**Profile 觸發的 spec 修正（2026-05-15 sandbox cProfile，200K synthetic RECTANGLE @ 4.0s = 50K rec/sec）：**

| Hot spot | self time % | 說明 |
|---|---|---|
| `OasisStream.read` + `_top_io` + `BufferedReader.read` | **29%** | 1.6M 次 `stream.read(1)` per-byte method call（兩層 Python wrapper + 一層 C） |
| `decode_unsigned_int` self | **18%** | 1.4M 次 Python while-loop, 每 byte 一個 `b = stream.read(1); byte = b[0]` |
| `_read_rectangle` body | 20% | info-byte 解、modal 寫、payload dict build、`_layer_filtered_out` |
| `iter_records` 主迴圈 self | 15% | elif dispatch + yield 機制 |
| 其他 (`_decode_xy` / `decode_signed_int` / `tell` / `maybe_pop`) | ~18% | |

原 spec「callback API 省 ~30% decode time」mechanism 對但 magnitude 高估 — payload dict + yield 合計實測**僅佔 5-7%**，真正瓶頸是 **per-byte stream indirection**（29%）+ `decode_unsigned_int` Python loop（18%）。純做 callback API 樂觀估只能 77.7s → ~73s，打不到 sub-plan 自己訂的 < 50s 目標。

**User 決議（2026-05-15）：** scope 擴大為 callback + byte-reader 雙管齊下，拆 M1.13.3a / M1.13.3b。callback signature 用 `cb(reader)`，caller 從 `reader._modal` 拉欄位（最簡 API，跟原 spec `(reader, modal_state)` 設計一致）。

#### M1.13.3a: Callback consume API (skip dict + yield)  [status: done (2026-05-15 sandbox, awaiting D2DB runtime verification)]

API/architecture 層：拿掉 hot loop 的 payload dict alloc + yield tuple，準備好 store 端的 consume entry point；單獨大概省 5-7%（dict + yield overhead 的實測比重），主要為 M1.13.3b 鋪路。

- [ ] 新 `OasisReader.consume(callbacks: dict[int, Callable[[OasisReader], None]])`:
  - Callback signature: `cb(reader)`；caller 從 `reader._modal` / `reader._current_cell` 直接拉欄位
  - Inner loop 純 `_decode_body → cb(self)`；無 dict alloc、無 yield
- [ ] 改 hot decoders (`_read_rectangle` / `_read_polygon` / `_read_path` / `_read_trapezoid` / `_read_ctrapezoid` / `_read_circle` / `_read_text` / `_read_placement`) 接 `build_payload: bool = True` 參數
  - `build_payload=True`：維持原行為，return dict（iter_records 用）
  - `build_payload=False`：跳過末段 dict 組裝，return `None`，原 bytes-reading + modal mutation 邏輯不動（consume 用）
- [ ] `iter_records` 維持現狀（內部隱性 `build_payload=True`），所有 113 streamer test 不動
- [ ] `OasisGeometryStore.run()` 改走 `consume`：每個 record_id 註冊一個 callback 直接 append 到 `_RectBuffer` / `_polys` / `_placements`
- [ ] **TestConsumeEquivalence**：同一份 OASIS 走 iter_records 路徑 vs consume 路徑，store output bit-identical（rect ndarrays / polygon list / placements list / cellnames map 全 deep-compare）

**驗證：**
- [x] `pytest tests/test_oasis_streamer.py tests/test_oasis_store.py tests/test_oasis_walker.py -q` → **163 passed**（160 baseline + 3 新 TestConsumeEquivalence）

#### M1.13.3b: bytearray-backed byte reader  [status: done (2026-05-15 sandbox, awaiting D2DB runtime verification)]

打掉真正的瓶頸 — per-byte `stream.read(1)` 的 method call indirection。

- [ ] `OasisStream` 新增 fast-path: 把當前 substream 攤平成 `bytes` (outer file mmap'd 或全讀進來；CBLOCK 解壓後本來就是 bytes) + 持 integer `pos`
- [ ] 新 module-level fast decoder：`decode_unsigned_int_buf(buf: bytes, pos: int) -> tuple[int, int]`（return new pos），避免 `stream.read(1)` 兩層 wrapper + bytes-of-len-1 alloc
- [ ] Hot decoders 改成接 `(buf, pos)` 走 fast path；非 hot path（START/END/PROPERTY/...）仍可走原 `OasisStream.read` API
- [ ] CBLOCK push/pop：push 把當前 `(buf, pos)` 推 stack、buf 換成解壓 bytes、pos=0；pop 還原（語意跟 OasisStream.push_cblock/maybe_pop_exhausted 對齊，但操作純 Python tuple stack）
- [ ] `iter_records` + `consume` 內部主迴圈都改吃 `(buf, pos)`；外部 API 不變
- [ ] `OasisReader.tell()` 仍可呼叫（從 active substream 起點 + pos 算）；error context（hex window）邏輯保留
- [ ] **TestByteReaderEquivalence**：CBLOCK / nested CBLOCK / multi-cell 各種 OASIS shape 比對 iter_records bit-identical pre/post refactor

#### M1.13.3c: Production-profile-driven hot-spot 重構  [status: planned]

**動機：** sandbox 預測 1.48× / 52.6s，production 實測 1.24× / 62.8s — gap 10s 來自 sandbox synthetic（single-cell pure-RECTANGLE）沒覆蓋到的 D2DB 特性：CBLOCK 壓縮塊、37 layer filter、multi-cell boundary、PLACEMENT repetition。要在純 Python 拿下剩 ~13s 需要先 profile 鎖定真正瓶頸。

- [x] 新 `tools/bench/profile_d2db_store.py`：cProfile harness 跑 `store.run()`、印 top-20 by self/cumtime + 8 個 targeted 子項（CBLOCK / RECTANGLE+POLYGON body / layer-filter / cell-boundary / PLACEMENT / uvarint hot loop / consume dispatch / numpy store append）的 self-time 百分比
- [ ] **User 規格機跑 D2DB partial 5M：** `python tools/bench/profile_d2db_store.py "<d2db>" --layer 17:101 --max-records 5000000 --out d2db_profile.txt`，貼回 stdout report + 上傳 `d2db_profile.txt`
- [x] **Profile 結果（2026-05-20, D2DB 5M, cProfile 88.98s）：** `decode_repetition` cumtime **45.1s（51%）**，type-1 grid 展開 listcomp（line 581）self **13.68s（15.4%）**；`_read_rectangle` 4.9M 呼叫只 keep 45,745（filter 砍 99.07%）；`read_uvarint` 17.47s self / 34.4M 呼叫；`decode_unsigned_int` 7.91s self / 24.3M 呼叫。targeted slice：repetition 15.2% / uvarint hot loop 23.5% / consume 19% / RECTANGLE body 7.1% / CBLOCK 1.7% / layer-filter 1.1%。

- [x] **No-regression**：177 OASIS test 全綠；wrapper 等價（type 1-5 offset byte-identical）；iter_records kept=展開 / filtered=strip 驗證。

## Risks / Open Questions

### 已知風險

1. **Cache invalidation 漏網**：mtime+size hash 對「內容相同但 mtime 變」會重跑（多餘但安全）；對「mtime 不變但內容改」會誤命中（罕見但危險）。Mitigation：Q&A phase 由 user 在 mtime+size vs sha256 之間擇一；schema version 欄位防 cache schema 升級時舊 entry 誤用。
2. **Walker vectorize 對 polygon 路徑的影響**：D2DB 0.02% polygon 罕用、不應拖累；vectorize 主要打 rect path，polygon 維持 list-of-ndarray 也 OK。已在 M1.13.2 註明。
3. **Fast-consumer 改 store 內部、API 表面看似不變但行為若有偏差很難 debug**：靠既有 9 個 store test + 新加的 `TestVectorizeEquivalence` 思路寫 `TestConsumeEquivalence`（同一檔走 iter_records vs consume，結果 bit-identical）保護。
4. **C ext (M1.13.4) Windows build tool**：Visual Studio Build Tools 安裝門檻；若啟動則 requirements 標 optional 且運行時 fall back。

### 待 user 後續確認

- M1.13.1 開工前的 Cache spec Q&A（存哪 / hash 策略 / 序列化 / eviction / version）
- M1.13.4 是否啟動（M1.13.3 跑完 benchmark 後決定）

### 外部依賴

- 必要：numpy (既有)、pytest (既有)
- Optional：cython / cffi（僅 M1.13.4 啟動時）
- 不引入：numba（為避免額外 cold start + 安裝複雜度）
